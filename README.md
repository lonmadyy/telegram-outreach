# Telegram Outreach

Автоматизация исходящих рассылок и приглашений в Telegram через несколько
userbot-аккаунтов, под управлением одного административного бота. Список адресатов
загружается TXT-файлом, сообщения собираются из шаблонов (spintax + переменные),
работу ведут параллельные воркеры с человекоподобными задержками и встроенной
защитой от банов.

> **Канонический документ проекта — [ARCHITECTURE.md](ARCHITECTURE.md).** Любые
> детали поведения и схемы данных описаны там; README — про запуск и эксплуатацию.

## Возможности

- **Два типа кампаний:** рассылка в личные сообщения (DM) и приглашение в канал/группу (invite).
- **Мультиаккаунтность:** пул воркеров, атомарная выдача задач (`SKIP LOCKED`), старт со смещением.
- **Антибан:** автопаузы при `FloodWait`/`PeerFlood`, адаптивное снижение лимита до 75%, quiet hours.
- **Warmup:** прогрев свежих аккаунтов (подписки, присутствие) с нулевыми лимитами в первые часы.
- **Мониторинг SpamBot:** периодический опрос @SpamBot, авто-снятие/назначение ограничений.
- **Дедупликация:** глобальный реестр обработанных адресатов между кампаниями.
- **Отчётность:** CSV-отчёты по кампаниям и выгрузка логов прямо из бота.
- **Эксплуатация:** скрипты резервного копирования и восстановления БД и сессий.

## Стек

Python 3.12 (async), Telethon (userbot), aiogram 3.x (управляющий бот),
PostgreSQL 16 + asyncpg + SQLAlchemy 2.0, Alembic, APScheduler, loguru, Docker Compose.

## Требования

- VPS: 2 vCPU, 2 GB RAM, 20 GB SSD, Linux (рекомендуются Ubuntu 22.04/24.04, Debian 12).
- Установленные Docker и Docker Compose.
- Открытый исходящий трафик к Telegram DC (`149.154.0.0/16`, `91.108.0.0/16`).
- Telegram API-ключи (`api_id`/`api_hash` с https://my.telegram.org) и токен бота от @BotFather.

## Установка и запуск

```bash
git clone <repo> telegram-outreach
cd telegram-outreach

# 1. Создать .env (см. раздел «Переменные окружения»), задать права 600.
nano .env
chmod 600 .env

# 2. Собрать и запустить.
docker compose up -d --build

# 3. Проверить статус и логи.
docker compose ps
docker compose logs --tail=100 app
```

Миграции БД (Alembic) применяются автоматически при старте приложения (§18.4).
Накатить вручную при необходимости:

```bash
docker compose exec app alembic upgrade head
```

Дальше — в Telegram от аккаунта из `ALLOWED_USER_IDS`: `/start` → `/add_account`
(FSM запросит телефон, код, при необходимости пароль 2FA) → `/new_campaign`.

## Переменные окружения (`.env`)

Секреты значениями ниже не приводятся — подставьте свои.

| Ключ | Назначение | Пример / дефолт |
|---|---|---|
| `TG_API_ID` | api_id с my.telegram.org | `1234567` |
| `TG_API_HASH` | api_hash с my.telegram.org | *(секрет)* |

| `ALLOWED_USER_IDS` | whitelist админов (через запятую) | `12345,67890` |
| `POSTGRES_DB` | имя БД | `outreach` |
| `POSTGRES_USER` | пользователь БД | `outreach` |
| `POSTGRES_PASSWORD` | пароль БД | *(секрет)* |
| `POSTGRES_HOST` | хост БД (в compose — `db`) | `db` |
| `POSTGRES_PORT` | порт БД | `5432` |
| `FERNET_KEY` | ключ шифрования секретов в БД (опционально) | *(секрет)* |
| `TZ` | таймзона | `Europe/Minsk` |
| `LOG_LEVEL` | уровень логирования | `INFO` |
| `SESSIONS_PATH` | путь к `.session` внутри контейнера | `/app/data/sessions` |
| `LOGS_PATH` | путь к файловым логам | `/app/data/logs` |

Поведение по умолчанию (дневные лимиты, интервалы, quiet hours, период warmup и т.д.)
имеет значения в коде и переопределяется через таблицу `settings` командой `/set`
(см. `/settings`). Полный список — в [ARCHITECTURE.md §11](ARCHITECTURE.md).

## Эксплуатация

### Резервное копирование

`backup.sh` создаёт дамп БД (`db_<ts>.sql.gz`) и архив сессий (`sessions_<ts>.tar.gz`),
удаляет копии старше 14 дней. Параметры берутся из `.env`. Ручной запуск:

```bash
./backup.sh
```

Cron ежедневно в 03:00:

```cron
0 3 * * * /opt/telegram-outreach/backup.sh >> /var/log/to-backup.log 2>&1
```

Переопределяемые переменные: `BACKUP_DIR` (по умолчанию `/var/backups/telegram-outreach`),
`RETENTION_DAYS` (по умолчанию `14`).

### Восстановление

```bash
./restore.sh --db /var/backups/telegram-outreach/db_<ts>.sql.gz
./restore.sh --sessions /var/backups/telegram-outreach/sessions_<ts>.tar.gz
# или сразу оба:
./restore.sh --db <...> --sessions <...>
```

Скрипт запросит подтверждение (перезапись данных). После восстановления:
`docker compose restart app`.

### Мониторинг

В v1 без внешних систем (§18.3):

```bash
docker compose ps                       # статус контейнеров
docker compose logs --tail=200 app      # текущие логи
```

О критических событиях (PeerFlood, dead-аккаунт, ошибка БД) бот уведомляет админа сам.
Историю можно выгрузить командой `/export_log`.

### Рестарт после падения

`restart: unless-stopped` в `docker-compose.yml` обеспечивает автоперезапуск. При старте
приложение ждёт healthcheck Postgres, накатывает миграции, возвращает зависшие задачи
`in_progress` (>1ч) в очередь и подхватывает кампании в статусе `running` (§18.4).

## Безопасность

- `.env` — в `.gitignore`, права `600`. Каталог `data/sessions/` — `700`, файлы `.session` — `600`.
- Доступ к боту — только `ALLOWED_USER_IDS` (middleware), остальные апдеты игнорируются молча.
- Postgres доступен только внутри docker-сети; на VPS — firewall (только SSH), вход по ключам.
- При компрометации `BOT_TOKEN` — `/revoke` в @BotFather, новый токен в `.env`, рестарт.
- `.session`-файлы дают полный доступ к аккаунтам — храните бэкапы в защищённом месте,
  рассмотрите шифрование диска (LUKS). Подробнее — [ARCHITECTURE.md §14](ARCHITECTURE.md).

## Траблшутинг

| Симптом | Причина | Что делать |
|---|---|---|
| Аккаунт `dead` | Сессия отозвана/забанена/деактивирована (401) | Реавторизовать через `/add_account`; для свежих аккаунтов не форсировать активность сразу |
| Аккаунт `pause` | `FloodWait` | Норма — аккаунт сам снимет паузу по истечении; кампания продолжится |
| Аккаунт `spam_blocked` | `PeerFlood` или ограничение от SpamBot | 12ч карантин + лимит 75% на 7 дней; снимется автоматически после ok от SpamBot |
| Кампания `paused` сама | Целевой чат недоступен / нет прав инвайта ни у одного аккаунта | Проверить чат и право «Добавление участников», затем `/resume <id>` |
| `/export_report` пустой | В кампании нет задач (все отфильтрованы дедупом) | Это валидно — в отчёте только сводка; проверьте `processed_clients` |
| `/export_log` «не найдены» | Нет логов за период (не было активности) | Указать другой период; логи копятся в `data/logs/` |
| `/export_log` «слишком большой» | Архив > 50 МБ (лимит Telegram) | Уменьшить `N` |
